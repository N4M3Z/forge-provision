Per-user runtime configurations that should sync across machines live in the **dotfiles** repo (`~/Developer/N4M3Z/dotfiles`), tracked by chezmoi. Source path follows the `dot_` prefix convention: `dot_config/<app>/config` deploys to `~/.config/<app>/config`. Examples that belong here: `ghostty`, `atuin`, `starship`, `revdiff`, `tmux`, `zsh`, `git`, `ssh`.

Machine-provisioning artifacts that bootstrap a fresh Mac live in **forge-provision**: `manifests/Brewfile`, `scripts/install/*.sh`, `scripts/configure/*.sh`, `scripts/migrate/*.sh`, ADRs in `docs/decisions/`. A configure script may run `chezmoi apply` to deploy a config, but the canonical source of the config still lives in dotfiles, not in `manifests/`.

Boundary test: "does this answer *what should my machine look like once it's set up?* or *how do I get a fresh machine to that state?*" The first goes to dotfiles. The second goes to forge-provision. When in doubt, the config probably belongs in dotfiles — provisioning concerns are about reaching the state, not maintaining it.

**Capture on touch.** Any session that modifies a per-user runtime config (a file under `~/.config/<app>/`, `~/.gitconfig`, `~/.zshrc`, `~/.ssh/config`, or any other dot-file in `$HOME`) must bring that config into dotfiles in the same session. Don't defer with "I'll capture it later" — the deferred state is where divergence happens. A fresh machine that runs `chezmoi apply` should reproduce exactly what the last session left behind. The pattern is: edit the chezmoi source (`dotfiles/dot_<path>`), run `chezmoi apply` to deploy, verify against the live file. If a configure script also writes to the same target (e.g., `scripts/configure/git-identity.sh` setting `user.name`), the dotfile and the script must agree on values; treat the dotfile as the canonical source and let the script become redundant over time.
